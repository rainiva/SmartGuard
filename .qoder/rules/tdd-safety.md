# TDD 开发规则（强制）

## 规则继承
本项目规则继承自用户级 `tdd-safety.md`，未覆盖部分以用户级为准。项目级规则可针对 SmartGuard 技术栈（C# / WPF / Windows Service）做补充约束。

## 0. 中文编码安全规则 —— 文件读写强制规范
- **禁止使用不稳定、不可靠的方式读取或修改文件。**
- 涉及中文内容的文件写入，必须使用显式 UTF-8（无 BOM）编码，禁止使用依赖系统默认编码的 API（如 PowerShell 的 `Set-Content` 无编码参数、`>>` 重定向等）。
- 推荐方式：使用 .NET 的 `[System.IO.File]::WriteAllText($path, $content, [System.Text.UTF8Encoding]::new($false))` 或等效显式编码 API。
- 禁止在终端命令中直接嵌入多行中文字符串执行，应优先使用项目内文件 + `Write` 工具，或确保终端编码为 UTF-8 后再执行。
- 文件读取时必须显式指定编码，避免依赖系统默认编码导致中文乱码。

## 适用场景
所有功能开发、缺陷修复、行为变更任务，无论改动大小。

## 规则正文

### 1. RED 先行 —— 测试先写，确认失败
- 必须先写/修改测试，运行并确认按预期失败。
- **未见失败，不得写任何生产代码实现。**
- 禁止以"优化、refactor、工具层、改动小、赶时间"等理由跳过 RED 阶段。

### 2. 已先写实现 —— 回退重来
- 若发现已先写了实现（无论有意无意），**必须删除实现**，从失败测试重新开始。
- **禁止事后补测糊弄。** 补写的测试不能替代先写测试的流程要求。

### 3. GREEN 最小实现
- 仅编写让当前失败测试通过的最小代码。
- 测试全绿后方可进入下一阶段。

### 4. REFACTOR 的前提
- **禁止在无失败测试的情况下，对生产代码做 extract/split 等批量改写。**
- 重构必须基于全绿测试基线，且重构后测试仍全绿。

### 5. 完成声明的验证义务
- **声称完成前：必须运行测试，并在回复中贴出通过输出。**
- 无测试输出截图/文本，不得标记任务完成。

### 6. 测试必须面向真实用户 UI 操作流程
- 测试必须模拟用户的真实操作：点击、输入、导航、观察可见反馈等。
- **禁止只测方法、函数、接口或内部实现细节。**
- 豁免场景：纯工具类/无UI的底层库（如日志格式化、配置解析），或已有端到端测试覆盖的模块。

## 阻断项速查
| 行为 | 判定 |
|------|------|
| 先写实现后补测试 | 违规，删除实现重来 |
| 未见测试失败就写实现 | 违规，先写测试并确认失败 |
| 无失败测试就批量重构生产代码 | 违规，先补测试或停止重构 |
| 声称完成但未贴测试通过输出 | 违规，运行测试并贴输出 |
| 测试只验证方法返回值/内部状态 | 违规，改为 UI 操作流程测试 |
